## 📌 产品定位

**AI 问数平台**是一个面向企业运营场景的 AI 数据产品（Text-to-Insight），核心目标是让运营人员、大区管理、集团高层通过**自然语言对话**直接获取业务数据结论，无需依赖数据分析师或 BI 报表。

系统自动处理三类核心问题：
- **指标口径歧义** — 统一定义 GMV、支付单量、AOV 等核心指标，全公司数字唯一可信
- **行级权限控制** — SQL 执行层强制过滤，集团/大区/公司三级隔离，模型不可绕过
- **AI 解释输出** — 每次回答强制携带时间口径、cutoff_time、权限范围，数字来龙去脉一目了然

## 🎯 解决的核心痛点

| 痛点 | 传统现状 | 本平台解法 |
|------|---------|-----------|
| **数据获取慢** | 运营提需求 → 数据组排期 → 3 天后出报表 | 自然语言输入，10 秒内返回结论 |
| **口径不一致** | GMV 在不同报表有不同定义 | 指标语义层统一口径，问数时自动调用 |
| **权限不可控** | BI 报表无法精细控制跨区访问 | SQL 执行前强制注入 `WHERE org_id IN (...)` |
| **数据不可解释** | 运营看到数字，不知道为什么高/低 | 每次回答附带 AI 解释 + 推荐追问方向 |
| **工具门槛高** | SQL / BI 工具需要专业培训 | 自然语言输入，无需任何技术门槛 |

## 💡 核心功能详解

### 1. AI 问数工作台 — TL;DR 四段式回答结构

每次 AI 回答包含四个固定区块，顺序不可变更：

```
┌─────────────────────────────────────────────────┐
│ 📊 TL;DR                                       │
│ 今日 GMV 142.3 万，较昨日同期 +8.2%                │
├─────────────────────────────────────────────────┤
│ 📈 主图表 / 指标卡                                │
│ [近 7 日 GMV 趋势折线图]  [今日 GMV: 142.3万]       │
├─────────────────────────────────────────────────┤
│ 📋 口径与范围说明                                 │
│ 时间口径：pay_time（支付时间）                      │
│ 数据截至：2026-03-06 10:00（cutoff_time）         │
│ 订单口径：支付成功订单数                            │
│ 权限范围：华东大区（上海/杭州/南京/苏州/宁波）          │
├─────────────────────────────────────────────────┤
│ 🔍 推荐追问                                      │
│ [按渠道拆分 GMV]  [看今日环比]  [为什么比BI低？]      │
└─────────────────────────────────────────────────┘
```

### 2. 行级权限模型 — 三级组织树

同一个问题「今天 GMV 多少？」，不同角色得到不同结果——这是**系统默认行为**，无需手动配置：

| 角色 | 权限范围 | 返回结果 |
|------|---------|---------|
| 集团 HQ（张总监） | 全量 | 全国所有大区汇总数据 |
| 华东大区（王运营） | 华东范围 | 上海/杭州/南京/苏州/宁波汇总 |
| 上海公司（李专员） | 本公司 | 仅上海公司数据，查杭州 → 直接拒绝 |

**权限注入机制（技术实现）：**
```sql
-- AI 生成的原始 SQL
SELECT SUM(pay_amount) - SUM(refund_amount) AS gmv
FROM fact_trade
WHERE dt = '2026-03-06'

-- 系统强制注入后（不可绕过）
SELECT SUM(pay_amount) - SUM(refund_amount) AS gmv
FROM fact_trade
WHERE dt = '2026-03-06'
  AND org_id IN ('SH001')  -- ← 由权限系统生成，模型无法修改
```

越权请求**不返回任何衍生值**（包括汇总、占比、排名），避免通过推算泄露越权数据。

### 3. 指标语义层 — MVP 首批核心指标

| metric_code | 指标名称 | 定义公式 | 时间字段 |
|-------------|---------|---------|---------|
| `gmv_pay` | GMV（支付口径） | `SUM(pay_amount) - SUM(refund_amount)` | pay_time |
| `paid_order_cnt` | 支付单量 | `COUNT(DISTINCT paid_order_id)` | pay_time |
| `aov` | 客单价 AOV | `GMV / 支付单量` | pay_time |
| `paid_user_cnt` | 支付用户数 | `COUNT(DISTINCT paid_user_id)` | pay_time |
| `refund_amount` | 退款金额 | `SUM(refund_amount)` | pay_time |

**治理规则：**
- **同义词词典（86 条）**：`成交额 / 销售额` → `GMV`，`下单量 / 订单数` → `paid_order_cnt`
- **歧义规则（18 条）**：`「上海」` → 优先匹配上海公司（组织维），城市维度二期支持
- **默认口径**：订单量默认 = 支付成功订单数，时间口径默认 = `pay_time`

### 4. 今日类问题口径硬规则

```
✅ 正确：今日 vs 昨日环比
   = 今日 00:00~cutoff_time  vs  昨日 00:00~cutoff_time  （同窗口对比）

❌ 禁止：今日截断值 vs 昨日全天
   → 会系统性低估增长率，误导运营决策

⚠️  每次回答必须展示 cutoff_time
   → 示例：数据截至 2026-03-06 10:00
```

## 🔄 问句治理飞轮

```
用户提问（失败）
      │
      ▼
失败分类分析
  ├── 歧义未消解  → 补澄清快捷选项        [优先级：高]
  ├── 指标未覆盖  → 补创建指标任务        [优先级：高]
  ├── 维度未支持  → 明确提示 + 路线规划   [优先级：中]
  └── 分析能力不足 → 引导追问替代直答    [优先级：中]
      │
      ▼
规则补充 / 指标扩展
      │
      ▼
问数成功率提升 → 用户信任增强 → 更多问数 → 更多治理数据
      └─────────────────────────────────┘（飞轮闭环）
```

## 🎨 设计规范

| 要素 | 规范值 | 说明 |
|------|--------|------|
| 主色 | `#3A78F2` | 数据蓝，企业级 SaaS 专业感 |
| 侧边栏 | `#0B1220` | 深夜蓝，强化内容聚焦 |
| 背景 | `#F5F7FA` | 冷调灰白，长时间使用低疲劳 |
| 圆角 | `18px`（卡片）/ `12px`（小元素） | 现代数字原生风格 |
| 字体 | `Inter` + `Noto Sans SC` + `JetBrains Mono` | 数字用 Mono，提升可读性 |

**信息架构四原则：**
1. **结论前置** — TL;DR 永远是第一个区块
2. **口径透明** — 每个数字旁边都有来源说明
3. **渐进披露** — Dashboard 摘要 → 追问详情 → 血缘溯源
4. **默认最优** — 时间口径、对比窗口、维度全部预设，减少用户决策成本

## 🚀 快速开始

```bash
# 克隆项目
git clone [https://github.com/your-org/ai-ask-data-platform.git
cd ai-ask-data-platform](https://github.com/panglezym/AI-Powered-Data-Query-Platform/tree/main)

# 安装依赖
npm install

# 配置环境变量
cp .env.example .env
# 编辑 .env，填写数据仓库连接、权限服务地址等配置

npm run dev

# 构建生产版本
npm run build
```

**环境要求：**
- Node.js ≥ 18
- 数据仓库：ClickHouse / Hive（任选其一）
- 权限服务：需对接企业组织树 API

## 🤝 贡献指南

欢迎提交 Issue 和 Pull Request。在贡献前请阅读：

- **指标新增**：需在 `指标中心` 配置 metric_code、定义公式、时间字段、维度，并同步更新同义词词典
- **权限变更**：任何涉及 org_id 过滤逻辑的修改必须经过安全审查，确保不引入越权漏洞
- **口径修改**：指标定义修改前，请通过`数据血缘`模块评估下游消费方影响范围
